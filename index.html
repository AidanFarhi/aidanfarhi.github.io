Software Design and Engineering üñ•Ô∏è

The application was built loosely following the Controller-Service-Repository pattern.

The Controller layer handles recieving and responding to requests, interacting with the Service layer.

The Service layer is where the bulk of the business logic resides, interacting with the Controller and Repository layer.

The Repository layer handles all interactions with the database. It interfaces with the Service layer.

Architecturally, it is a monolith hosted on Heroku that has a separate DB instance provisioned using Neon.

Here is the tech stack that I used:

  • Golang
  • PosgreSQL
  • JavaScript
  • HTML
  • CSS
  • Heroku
  • Neon

I also made sure to write unit tests for the service layer. By doing so, I validated that the core business logic is functioning as expected. I achieved 95% code coverage for the service package.

One of the course outcomes was this:

Develop a security mindset that anticipates adversarial exploits in software architecture and designs to expose potential vulnerabilities, mitigate design flaws, and ensure privacy and enhanced security of data and resources

I was able to demonstrate growth in this area by ensuring that all user interactions with the application are secured by session-based authorization. I also made sure that all passwords are encrypted before being stored in the database.

Algorithms and Data Structures ‚öôÔ∏è

While building out the application, I leveraged my skills in Algorithms and Data Structures.

This aligns with the following course outcome:

Design and evaluate computing solutions that solve a given problem using algorithmic principles and computer science practices and standards appropriate to its solution, while managing the trade-offs involved in design choices

On the backend, one of the core structures is the WeightEntry struct, which represents a single database row and is collected into Go slices ([]WeightEntry) that serve as the primary in-memory containers. These are dynamic arrays that grow amortized O(1) on append.

In the database layer, PostgreSQL handles the heavy work. The ROW_NUMBER() window function uses internal sorting (O(n log n)) over partitions to deduplicate entries by date, while B-tree indexes support efficient lookups on filters and ORDER BY clauses. On the frontend, the fetched weight entries become arrays of plain JavaScript objects, which are then sorted with an O(n log n) comparison sort, mapped to labels and values in O(n), and scanned for min/max in O(n).

Chart.js ultimately transforms these arrays into its own typed arrays and drawing instructions for canvas rendering.

Across the whole app, the dominant structures are dynamic arrays, structs, and indexed relational tables, and the main algorithms involve sorting, window functions, and linear array transformations.

Databases üõ¢Ô∏è

The PostgreSQL database is hosted on Neon, which provides a fast, cloud-native platform with built-in scalability and branching capabilities. Each table in the schema plays a clear role in maintaining application integrity. I used the pgx library to ensure efficient, type-safe communication between the Go server and the database, allowing for clean queries, robust error handling, and high performance across all data operations.

Here are the tables and their descriptions:

  • user_account - stores core identity and authentication data.
  • user_session - tracks active sessions to support secure, cookie-based login flows.
  • weight_entry - captures all user-submitted weight records for analysis and display.
